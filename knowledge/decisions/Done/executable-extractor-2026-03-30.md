Read the scanner QA report at `anvil/knowledge/qa/scanner-qa-2026-03-30.md` for live counts (939 files registered for invoice-pulse). Also read Forge's `forge/src/extractor.py` (first 50 lines) for pattern reference. **Task:** Design the extractor blueprint covering two components:
> **(A) `src/parsers/python_parser.py`** — Python AST-based source parser. Must specify: (1) `parse_file(file_path)` — reads a .py file, parses with `ast.parse()`, returns a list of chunk dicts ready for DB insertion. Each chunk dict contains: name, chunk_type, content (full source text of the function/class/method), content_hash (SHA-256), start_line, end_line, parent_name (for methods — the class name, None for top-level). (2) Chunk type mapping from AST nodes: `ast.FunctionDef` and `ast.AsyncFunctionDef` at module level → 'function', `ast.ClassDef` → 'class', `ast.FunctionDef` inside a class → 'method', test functions (name starts with `test_` or `Test`) → 'test_case'. (3) How to extract source text per node — use `ast.get_source_segment()` or line-number slicing from the original file content. (4) `extract_symbols(file_path, ast_tree)` — walk the AST and return lists of: imports (from/import statements with module names), function/method calls (ast.Call nodes with resolved names where possible), class definitions (name + bases for inheritance), test function → module mapping (test file naming convention `test_*.py` → tested module). (5) `compute_structural_metadata(ast_node)` — per-chunk metadata: cyclomatic complexity (count of if/elif/for/while/try/except/with/and/or/assert nodes), nesting depth (max depth of control flow nesting), parameter count (for functions/methods), import count (imports within the chunk), has_docstring (boolean), line_count. (6) Special: SQLAlchemy pattern recognition — detect classes inheriting from `db.Model` or `Base`, detect `db.Column()`, `db.relationship()` calls, annotate these in symbol_bindings with binding_type 'defines' and a descriptive symbol_name like `table:carrier_profiles` or `column:carrier_profiles.name`.
> **(B) `src/extractor.py`** — orchestrator that processes all files needing extraction. Must specify: (1) `extract_project(conn, project_name, cycle_id)` — main entry point. Queries `code_chunks` for all file-level module chunks for this project, reads each file, runs python_parser on it, replaces the file-level module chunk with function/class/method-level chunks (or keeps the module chunk and adds sub-chunks — design decision needed). Creates symbol_bindings and dependencies. Returns summary dict. (2) Chunk replacement strategy — the scanner created file-level 'module' chunks as placeholders. The extractor must decide: (a) delete the module chunk and insert function/class/method chunks, or (b) keep the module chunk as a parent and insert sub-chunks with parent_chunk_id pointing to it. Option (b) preserves the file-level view while adding granularity — recommend this. (3) Dependency construction — after all chunks for a project are extracted, run a resolution pass: for each import binding, try to match the imported module/symbol to an existing chunk. If found, create a chunk_dependency edge. Scope: within_file if both chunks share file_path, cross_file otherwise. For call bindings, same resolution. For inheritance, match class bases to class chunks. (4) MinHash fingerprinting — after extraction, compute MinHash signatures for each chunk's content using datasketch. Store in chunk_fingerprints. Then run similarity detection: compare each new chunk's MinHash against all existing chunks in the project, store pairs above threshold in chunk_similarities. (5) Structural metadata storage — the blueprint should decide where structural_metadata lives. The schema has no explicit structural_metadata table or column on code_chunks. **Design decision needed:** add a `structural_metadata` TEXT column to code_chunks (JSON blob, matching study's pattern), or create a separate `structural_metadata` table. Adding a column is simpler and follows study's approach; a separate table allows independent updates. Recommend column addition with a schema migration. **Include "How to verify" section:** specify that QA should verify against known invoice-pulse structure — e.g., `engines/validator.py` should produce chunks for each gate function, `database.py` should produce SQLAlchemy model chunks, test files should produce test_case chunks. **Deposit:** `anvil/knowledge/architecture/extractor-blueprint-2026-03-30.md` using `with open()`.
