Read `anvil/src/db.py` for `create_cycle_report()` and `get_cycle_report()`. **Task:** Design the Lab blueprint. The Lab is where scored data becomes actionable intelligence. It reads health_scores, chunk_dependencies, chunk_similarities, and git_changes, then produces findings in formats the Planner and agents consume. Must specify:
> **(1) `run_lab(conn, project_name, cycle_id)`** — main entry point. Runs all analysis functions, assembles the cycle report, deposits findings as markdown. Returns summary dict. **(2) Finding types** — the Lab must produce these specific finding categories: **(a) Coverage gaps** — chunks with composite_score > threshold AND coverage_score > 0.8 (untested). Output: list of function names, file paths, and risk scores. These become Planner constraints: "add test coverage before modifying these functions." **(b) Coupling hotspots** — chunks with coupling_score > threshold AND high inbound dependency count. These are modules where changes cascade. Output: chunk name, file path, inbound/outbound counts, list of dependents. These become Planner constraints: "verify these N dependents when modifying this module." **(c) Clone candidates** — chunk_similarities pairs above threshold. Output: pairs with file paths, names, similarity scores, diff size estimate. These become Planner fix executables: "consolidate these duplicate functions." **(d) Staleness alerts** — chunks with staleness_score > threshold. These are functions whose dependencies have been updated but the function itself hasn't. Output: chunk name, last modified date, stale dependencies with their modification dates. These become investigation prompts. **(e) Complexity hotspots** — chunks with complexity_score > threshold. Output: function name, cyclomatic complexity, nesting depth, parameter count. These become refactoring candidates. **(f) Co-change patterns** — files that frequently change together in git_changes (appear in the same commits). Output: file pairs with co-change count. These reveal implicit coupling not captured by import/call dependencies.
> **(3) Output formats** — specify two output modes: **(a) Cycle report markdown** — deposited to `anvil/knowledge/research/cycle-N-findings-YYYY-MM-DD.md`. Human-readable report with sections per finding type, sorted by risk. This is what the CEO reads. Must include: executive summary (total chunks, risk distribution, top 5 riskiest), then sections for each finding type with actionable items. **(b) cycle_reports DB row** — summary stats stored via `db.create_cycle_report()` for historical tracking across cycles. **(4) Specialist file update data** — the Lab must produce a section with current structural metrics that specialist sync prompts can reference: total functions/classes/methods, test count, table count (from SQLAlchemy detection), file count, top-10 most complex functions, top-10 most coupled modules. This data replaces stale numbers in specialist files. **(5) Threshold configuration** — all thresholds must be configurable in config.py: `HIGH_RISK_THRESHOLD` (composite score above which a chunk is flagged), `COVERAGE_GAP_THRESHOLD` (coverage score above which a chunk is considered untested), `COUPLING_HOTSPOT_THRESHOLD`, `STALENESS_THRESHOLD`, `COMPLEXITY_THRESHOLD`, `SIMILARITY_THRESHOLD` (for clone detection), `COCHANGE_MIN_COUNT` (minimum number of co-commits to flag a pair). **(6) `generate_planner_constraints(conn, project_name, cycle_id)`** — produces a structured list of constraints the Planner should include in future plans for this project. Format: each constraint has a type (coverage_required/verify_dependents/investigation_needed/refactor_candidate), target (file_path + function name), reason (why this constraint exists), and severity (high/medium/low). This is the direct bridge between Anvil findings and Planner orchestration. **Design decisions:** (1) Should the Lab also produce executable plans directly, or just findings that the Planner turns into plans? Recommend findings only — the Planner is the planning authority, not Anvil. (2) Should co-change analysis use raw commit overlap or normalized Jaccard similarity? Jaccard is more meaningful for files with different change frequencies. **Include "How to verify" section:** cycle report exists and contains all sections, DB row created, thresholds produce reasonable finding counts against the known 29 high-risk chunks. **Deposit:** `anvil/knowledge/architecture/lab-blueprint-2026-03-30.md` using `with open()`.
