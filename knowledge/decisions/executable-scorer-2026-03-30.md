Read `anvil/src/db.py` for the available CRUD functions (`create_health_score`, `create_test_result`, `get_chunks_by_project`, `get_dependencies`, `get_bindings_by_chunk`). **Task:** Design the scorer blueprint. The scorer reads data from 5 tables (code_chunks, chunk_dependencies, chunk_symbol_bindings, git_changes, structural_metadata column on code_chunks) and writes to health_scores. Must specify:
> **(1) `score_project(conn, project_name, cycle_id)`** — main entry point. Loads all chunks for the project, computes 5 dimension scores per chunk, computes composite score, inserts health_score rows. Returns summary dict (chunks_scored, score_distribution: {high_risk: N, medium: N, low: N}). **(2) Volatility formula** — for each chunk, query git_changes for the chunk's file_path. Compute: change_count (number of commits touching this file in the git history window), recency_weight (more recent changes weight higher — e.g., linear decay from 1.0 for today to 0.0 for GIT_HISTORY_WEEKS ago), co_change_count (how many other files changed in the same commits). Normalize to 0.0–1.0 range. Specify the exact normalization approach (e.g., min-max across all chunks in the project, or percentile-based). **(3) Coverage formula** — for each chunk, count test_case chunks that have a 'tests' binding targeting this chunk (via chunk_symbol_bindings). Coverage_score = 1.0 - (test_binding_count > 0 ? some_positive_value : 0.0). Chunks with zero test bindings should score highest risk (1.0 = worst). Specify whether this is binary (tested vs untested) or graded (more tests = better). **(4) Complexity formula** — read the chunk's structural_metadata JSON. Extract cyclomatic_complexity, nesting_depth, parameter_count. Combine into a single score. Specify weights and normalization. **(5) Coupling formula** — count inbound + outbound edges from chunk_dependencies for this chunk. Higher coupling = higher risk. Specify normalization. **(6) Staleness formula** — for each chunk, find its last commit date in git_changes. Find the most recent commit date among all its dependencies' files. If any dependency was modified more recently than the chunk itself, the chunk is stale. Staleness_score scales with how many dependencies are newer and how much newer they are. **(7) Composite score formula** — weighted combination of the 5 dimensions. Specify default weights (configurable in config.py). Recommend weights based on what signals are most predictive of bugs (from PROJECT_BRIEF: schema drift is #1 source of production crashes, parallel implementations diverging is #2). **(8) Test result ingestion** — specify `ingest_test_results(conn, project_name, cycle_id)` that runs `pytest` against the target project and parses output into test_results table. This populates the data that coverage scoring reads. **Design decisions:** (1) Should scoring be file-level or chunk-level? Volatility data is per-file (git_changes tracks files, not functions). Recommend file-level volatility propagated to all chunks in that file. (2) How to handle chunks with no git history (new files)? Recommend neutral score (0.5) not extreme. (3) Should test_results store individual test names or just counts? The schema has failed_test_names as JSON — use it for debugging, but scoring only needs pass/fail counts. **Include "How to verify" section** with specific checks: top-10 highest-risk chunks should be queryable, all chunks should have scores, composite should be in 0.0–1.0 range. **Deposit:** `anvil/knowledge/architecture/scorer-blueprint-2026-03-30.md` using `with open()`.
