## STEP 1 — ANVIL SYSTEMS ANALYST (scanner blueprint)

---

> You are the Anvil Systems Analyst. Read your specialist file at `anvil/agents/ANVIL_SYSTEMS_ANALYST.md` and the domain glossary at `anvil/knowledge/research/domain-glossary.md` first. Read `anvil/src/config.py` for current SCAN_TARGETS, EXCLUDED_DIRS, EXCLUDED_EXTENSIONS, and GIT_HISTORY_WEEKS. Read Forge's `forge/src/scanner.py` (first 80 lines) for the file discovery pattern — Forge walks knowledge directories and classifies .md files; Anvil walks source directories and classifies code files. **Task:** Design the scanner blueprint. The scanner has two jobs: (1) **File discovery + change detection** — walk a project directory, skip excluded dirs/extensions, compute SHA-256 content hash per file, compare against existing `chunk_fingerprints` to identify new/changed files since last cycle. Return a list of files needing extraction. (2) **Git history ingestion** — parse `git log` output for the last N weeks (from config.GIT_HISTORY_WEEKS), populate `git_changes` table with per-file change records.
> **Blueprint must specify:** (1) `scan_project(conn, project_name)` — main entry point. Resolves project path from config.SCAN_TARGETS, calls discover_files and ingest_git_history, returns a summary dict with counts (files_total, files_new, files_changed, files_unchanged, git_commits_ingested). (2) `discover_files(project_path, excluded_dirs, excluded_extensions)` — recursive directory walk, returns list of dicts with file_path, relative_path, extension, size_bytes. (3) `compute_file_hash(file_path)` — SHA-256 of file content, returns hex digest string. (4) `detect_changes(conn, project_id, discovered_files)` — for each file, compute hash, compare against most recent `chunk_fingerprints` entry for that file_path. Categorize as new (no prior fingerprint), changed (hash differs), or unchanged. Return three lists. (5) `ingest_git_history(conn, project_id, project_path, weeks)` — run `git log --numstat --format=...` for the last N weeks, parse output, insert into `git_changes` table. Specify the exact git log format string to use. Handle dedup: if a commit_hash already exists in git_changes for this project, skip it (idempotent). (6) `register_file_chunks(conn, project_id, new_and_changed_files, cycle_id)` — for new/changed files, create a file-level entry in `code_chunks` (chunk_type='file') with the content hash. This is a placeholder — the Extractor in Phase 3 will add function/class/method-level chunks. The file-level chunk ensures every scanned file is tracked even before extraction runs. **Design decisions:** (1) Should the scanner read file content into memory for hashing, or stream it? For invoice-pulse file sizes (all under 1MB), reading fully is fine. (2) Git log format — recommend `--format=%H|%aI|%an|%s` (hash, ISO date, author, subject) with `--numstat` for per-file additions/deletions. (3) Should git_changes track per-file stats (lines added/removed) or just the file list per commit? Per-file stats enable better volatility scoring in Phase 4. **Include "How to verify this was implemented correctly" section.** **Deposit:** `anvil/knowledge/architecture/scanner-blueprint-2026-03-30.md` using `with open("/Users/marklehn/Desktop/GitHub/anvil/knowledge/architecture/scanner-blueprint-2026-03-30.md", "w") as f: f.write(content)`.
